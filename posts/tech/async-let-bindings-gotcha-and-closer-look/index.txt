13:T48e0,<p>Swift 5.5 introduced a powerful set of concurrency tools to deal with asynchronous tasks, eliminating the complexity of callbacks and queues.</p>
<p>Besides Task and TaskGroup for managing child tasks, the <code>async let</code> syntax allows you to start concurrent tasks and bind their results to variables, following <a href="https://github.com/swiftlang/swift-evolution/blob/main/proposals/0304-structured-concurrency.md">structured concurrency</a> principles.</p>
<p>Yet, be cautious when writing <code>async let</code> bindings. Consider these two declarations below.</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">async</span> <span class="hljs-keyword">let</span> x <span class="hljs-operator">=</span> taskX() <span class="hljs-comment">// returns an Int</span>
<span class="hljs-keyword">async</span> <span class="hljs-keyword">let</span> y <span class="hljs-operator">=</span> taskY() <span class="hljs-comment">// retunns a String</span>
<span class="hljs-keyword">let</span> result <span class="hljs-operator">=</span> <span class="hljs-keyword">await</span> (x, y)
</code></pre>
<pre><code class="hljs language-swift"><span class="hljs-keyword">async</span> <span class="hljs-keyword">let</span> (x, y) <span class="hljs-operator">=</span> (taskX(), taskY())
<span class="hljs-keyword">let</span> result <span class="hljs-operator">=</span> <span class="hljs-keyword">await</span> (x, y)
</code></pre>
<p>At first glance, they seem equivalent. Both deliver <code>result</code> as a tuple of values from <code>taskX()</code> and <code>taskY()</code>. But are they truly the same? <strong>No, they’re NOT</strong>.</p>
<p>The difference lies in <em>concurrency</em>. In the first example, <code>taskX()</code> and <code>taskY()</code> run concurrently. That results in a total of 1s if each task takes 1s. Meanwhile, the two tasks in the second run sequentially, taking 2s in total. What a performance gap for such a subtle syntax change!</p>
<h2 id="why-the-difference">Why the Difference?</h2>
<p>The root cause is how Swift interprets the right-hand side of a binding. In the second example, <code>(taskX(), taskY())</code> is treated as a single asynchronous task. This task consists of two asynchronous calls which are performed <strong>one-by-one</strong> according to the <em>evaluation order</em>, from left to right.</p>
<p>I guess this is <strong>by design</strong> rather than an implementation limitation. Why? Let’s reverse-think it. If <code>(taskX(), taskY())</code> spawns concurrent tasks, how would you express a tuple of <em>sequential</em> tasks? And what about literal arrays like <code>[taskX(), taskY()]</code> - should they also imply concurrency? These not only clash with Swift’s left-to-right evaluation order but also lead to error-prone code.</p>
<p>If you want a shorthand for concurrent tuples, you may need a helper function like this:</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">func</span> <span class="hljs-title function_">asyncTuple</span>&#x3C;<span class="hljs-type">U</span>, <span class="hljs-type">V</span>>(<span class="hljs-keyword">_</span> <span class="hljs-params">first</span>: <span class="hljs-keyword">@autoclosure</span> () <span class="hljs-keyword">async</span> -> <span class="hljs-type">U</span>, <span class="hljs-keyword">_</span> <span class="hljs-params">second</span>: <span class="hljs-keyword">@autoclosure</span> () <span class="hljs-keyword">async</span> -> <span class="hljs-type">V</span>) <span class="hljs-keyword">async</span> -> (<span class="hljs-type">U</span>, <span class="hljs-type">V</span>) {
  <span class="hljs-keyword">async</span> <span class="hljs-keyword">let</span> firstValue <span class="hljs-operator">=</span> first(), secondValue <span class="hljs-operator">=</span> second()
  <span class="hljs-keyword">return</span> <span class="hljs-keyword">await</span> (firstValue, secondValue)
}

<span class="hljs-keyword">async</span> <span class="hljs-keyword">let</span> result <span class="hljs-operator">=</span> asyncTuple(taskX(), taskY())
<span class="hljs-keyword">async</span> <span class="hljs-keyword">let</span> (x, y) <span class="hljs-operator">=</span> asyncTuple(taskX(), taskY())
<span class="hljs-operator">...</span>
</code></pre>
<h2 id="a-closer-look">A Closer Look</h2>
<h3 id="async-let-vs-task">Async let vs. Task</h3>
<p>It’s important to know why <code>async let</code> is needed when we already have <code>Task</code>. In fact, the concurrent tuple in the previous section can be implemented this way:</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">let</span> tx <span class="hljs-operator">=</span> <span class="hljs-type">Task</span> { <span class="hljs-keyword">await</span> taskX() }
<span class="hljs-keyword">let</span> ty <span class="hljs-operator">=</span> <span class="hljs-type">Task</span> { <span class="hljs-keyword">await</span> taskY() }
<span class="hljs-keyword">let</span> result <span class="hljs-operator">=</span> <span class="hljs-keyword">await</span> (tx.value, ty.value)
</code></pre>
<p>Well, this touches on the topic of <a href="https://github.com/swiftlang/swift-evolution/blob/main/proposals/0304-structured-concurrency.md">structured concurrency</a> which we will not go into detail about in this post. While <code>Task</code> works, it’s <em>unstructured</em> in some cases. Tasks can outlive their scope (eg. with <code>Task.detached</code>) which breaks the rules of structured concurrency. Meanwhile, <code>async let</code> ensures execution within a defined scope (eg. within the function), allowing the compiler to enforce compile-time checks for concurrency practices.</p>
<h3 id="async-let-implementation">Async let Implementation</h3>
<p>Syntactically, <code>let</code> and <code>async let</code> look similar, but their implementations differ. A regular <code>let</code> binding has two key phases: <em>instantiation</em> and <em>destruction</em>. An <code>async let</code> has one more: when the value is <strong>awaited</strong>, or ready. In Swift’s concurrency model, this can be roughly understood as a <strong>suspension point</strong> where the current execution gives up its thread for others to work (until gaining it back - alongside the result).</p>
<p>Consider this simple code and look at its SIL (Swift Intermediate Language) to understand the lifetime of an <code>async let</code> variable.</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">async</span> <span class="hljs-keyword">let</span> x <span class="hljs-operator">=</span> taskX() <span class="hljs-comment">// async -> Int</span>
