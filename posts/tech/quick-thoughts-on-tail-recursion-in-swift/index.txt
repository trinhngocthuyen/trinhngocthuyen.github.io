13:T16b1,<h3 id="problem">Problem</h3>
<p>I always thought that <em>Tail call optimization (TCO)</em>, sometimes called <em>tail recursion optimization</em>, is supported in most languages by default. It turns out to be opposite.</p>
<p>I happened to find it out when solving a Hackerrank problem (using Python). As a fan of functional programming, I used recursion for better readability. Of course, I wrote the code in <em>tail recursion</em> to avoid memory issues and let the system take care the rest. Yet, a <code>segmentation fault</code> exception was thrown as a potential evidence of the absence of TCO.</p>
<p>After a few minutes googling, I found that there is an approach to eliminate the memory issues without changing my code to while-loop style. Here is the <a href="http://chrispenner.ca/posts/python-tail-recursion">reference</a>. You don't need to understand it, just remember that in Python, there exist a solution to fix it.</p>
<p>How about Swift?</p>
<h3 id="investigation">Investigation</h3>
<p>Consider this code:</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">func</span> <span class="hljs-title function_">sumPrefix</span>(<span class="hljs-keyword">_</span> <span class="hljs-params">n</span>: <span class="hljs-type">Int</span>, <span class="hljs-keyword">_</span> <span class="hljs-params">acc</span>: <span class="hljs-type">Int</span>) -> <span class="hljs-type">Int</span> {
  <span class="hljs-keyword">if</span> n <span class="hljs-operator">==</span> <span class="hljs-number">0</span> { <span class="hljs-keyword">return</span> acc }
  <span class="hljs-keyword">return</span> sumPrefix(n <span class="hljs-operator">-</span> <span class="hljs-number">1</span>, acc <span class="hljs-operator">+</span> n)
}
<span class="hljs-keyword">_</span> <span class="hljs-operator">=</span> sumPrefix(<span class="hljs-number">1000000</span>, <span class="hljs-number">0</span>)
</code></pre>
<p>Build this code with <code>-Onone</code> (no optimization), and run the program. You will get a crash!</p>
<p><code>xcrun swiftc -Onone main.swift; ./main</code></p>
<p>When turning on the optimization, the program executes properly.</p>
<p><code>xcrun swiftc -O main.swift; ./main</code></p>
<p>Clearly, the tail recursion optimization was not supported in the <code>-Onone</code> build. Otherwise, it would have not crashed. About the <code>-O</code> build, the tail call was optimized. A nice way to inspect it is looking at the <code>asm</code> file: <code>xcrun swiftc -O -S main.swift > main.asm</code>.</p>
<p>No <code>callq</code> instruction was found! That means, the recursive instruction which expands the stack frame is replaced by jump instructions.</p>
<p>But wait! No <code>callq</code> indicates that the invocation <code>_ = sumPrefix(1000000, 0)</code> was inlined. How do we know that the exception was not raised as a result of inlining or TCO? Let's force Swift not to inline this function:</p>
<pre><code class="hljs language-swift"><span class="hljs-meta">@inline</span>(never)
<span class="hljs-keyword">func</span> <span class="hljs-title function_">sumPrefix</span>(<span class="hljs-keyword">_</span> <span class="hljs-params">n</span>: <span class="hljs-type">Int</span>, <span class="hljs-keyword">_</span> <span class="hljs-params">acc</span>: <span class="hljs-type">Int</span>) -> <span class="hljs-type">Int</span> {
	<span class="hljs-operator">...</span>
}
</code></pre>
<p>Now, <code>callq</code> is back! And there is only one function call.</p>

<h3 id="discussion">Discussion</h3>
<p>Although the problem does not happen in optimized builds, I expect the TCO support to be available in any level of optimization. You cannot write readable code that crashes on Debug builds.</p>
<p>As recursion is a crucial piece of functional programming, I believe this is definitely a lack of support and a limitation of language. Take Scala as an example, you could instruct the program to optimize the tail call using <a href="https://www.scala-exercises.org/scala_tutorial/tail_recursion">@tailrec</a>. Or at least, in Python, we could fix it with <em>decorators</em> and achieve the same result.</p>
<p>In fact, there was a proposal for TCO. You could find it <a href="https://lists.swift.org/pipermail/swift-evolution/2015-December/000370.html">here</a>. But it was a pretty long time ago... And I think it will keep staying there for a while.</p>
