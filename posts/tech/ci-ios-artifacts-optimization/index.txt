13:T327a,<h2 id="1-introduction">1. Introduction</h2>
<h3 id="11-pipeline-time-improvement">1.1. Pipeline Time Improvement</h3>
<p>As engineers, we always want to land our change on master as quickly as possible. Apart from the time it takes to resolve code review comments from peers, there's one constraint engineers have to face before getting their changes merged. That constraint is the <strong>CI pipeline time</strong> (ie. the time it takes for a CI pipeline to run against certain changes).</p>
<p>For iOS development, a typical CI pre-merge pipeline usually involves building the project, then running unit tests and UI tests. Therefore, the engineering work to reduce pipeline time can be broken down into 2 major problems: <em>build time improvement</em> and <em>test time improvement</em>. While build time improvement is a classic problem that is tackled by many initiatives (mostly driven by community's efforts), there are still many rooms for test time improvement.</p>
<h3 id="12-separation-of-build-and-test-jobs-in-ios">1.2. Separation of Build and Test Jobs in iOS</h3>
<p>I once shared a tip to utilize test parallelism within one single CI job (see: <a href="https://engineering.grab.com/tackling-ui-test-execution-time-imbalance-for-xcode-parallel-testing">Tackling UI tests execution time imbalance for Xcode parallel testing</a>). Regarding parallelism among CI jobs, it can be done by splitting tests into smaller sets and run them in multiple CI jobs. As following is the form of pipelines we desire to have:</p>
<p><img src="/images/ci/pipeline.png" alt=""></p>
<p>However, the separation of build and test jobs in iOS is not that straightforward because of 2 prominent factors:</p>
<p><strong>(F1)</strong> It's not feasible to obtain the list of tests we have at compile time. We can only use heuristic approaches to extract such info.</p>
<p><strong>(F2)</strong> Running unit/UI tests requires some build products from the <code>build</code> job. This is a <em>hard constraint</em> for platforms with statically-typed languages. What makes it harder for iOS development is that the iOS build artifacts are relatively big. Unfortunately, the size of artifacts we pass from one job to another is constrained by the size limit set by the CI infra.<br>
The growth in size of an iOS project is propotional to not only <em>the number of code lines</em> but also <em>the number of targets</em> we have.</p>
<ul>
<li>As of Oct 2020, there are 340 targets in our project, including 323 pod targets (aka. framework targets) and 17 executable targets (1 app target + 16 test targets). The artifacts size of our project is around 3.6GB which exceeds the size limit.</li>
</ul>
<p>This post introduces a tip to overcome the artifacts size constraint in (F2).</p>
<h2 id="2-a-closer-look-at-ios-build-artifacts">2. A Closer Look at iOS Build Artifacts</h2>
<p>Now, let's take a look at a project that has both <em>hosted test targets</em> and <em>non-hosted test targets</em>. Assume we're using CocoaPods to manage dependencies in your project. When building the project, the build products folder would be in the following structure.</p>
<pre><code class="hljs language-lua">|<span class="hljs-comment">-- App.app / -- App (*)</span>
|           |
|           |<span class="hljs-comment">-- Frameworks / -- DynamicFW_A.framework</span>
|           |               |<span class="hljs-comment">-- DynamicFW_B.framework / -- DynamicFW_B.bundle</span>
|           |
|           |<span class="hljs-comment">-- Plugins / -- HostedTestTarget.xctest</span>
|           |
|           |<span class="hljs-comment">-- StaticFW_C.bundle</span>
|
|<span class="hljs-comment">-- NonHostedTestTarget.xctest</span>
</code></pre>
<p>The <code>.app</code> bundles and <code>.xctest</code> bundles are both <strong>executable bundles</strong>. They have the same folder structure. Inside such a bundle:</p>
<ul>
<li>The <code>Frameworks</code> folder contains the <code>.framework</code> bundles of frameworks built as dynamic. Note that some system dynamic libraries such as <code>libswiftCore.dylib</code>, <code>libswiftFoundation.dylib</code>... also reside in this folder.</li>
<li>The <code>Plugins</code> folder contains the <code>.xctest</code> bundles of the app's hosted test targets (ie. test targets that use this app as its host app). Meanwhile, <code>.xctest</code> bundles of non-hosted test targets reside in the same folder of the <code>.app</code> bundles.</li>
<li>The resources and resource bundles (in form of <code>.bunlde</code> packages) of static frameworks are located under the root of the executable bundles.</li>
</ul>
<h2 id="3-duplicated-contents">3. Duplicated Contents</h2>
<h3 id="31-dynamic-frameworks">3.1. Dynamic Frameworks</h3>
<p>From the structure above, we can easily spot the duplication that if a dynamic framework is used in both the app and a test target, that framework exists in 2 places:</p>
<ul>
<li>Under the <code>Frameworks</code> folder of the <code>.app</code> bundle</li>
<li>Under the <code>Frameworks</code> folder of the <code>.xctest</code> bundle.</li>
</ul>
<p>In general, if a framework is used in N targets, it appears N times in the executable bundles. If we check the <strong>checksums of the frameworks</strong> in those executable bundles, they are all <strong>identical</strong>.</p>
<p>To understand why this happens, we can take a look at how CocoaPods integrates frameworks to the project.</p>
<p>For dynamic frameworks added to a target, CocoaPods adds a build phase called <code>[CP] Embed Pods Frameworks</code> at the end of the target build phases.</p>
<p><img src="/images/ci/cp_build_phase.png" alt=""></p>
<p>This build phase actually executes a script <code>"${PODS_ROOT}/Target Support Files/Pods-App/Pods-App-frameworks.sh"</code> to copy all dynamic frameworks (managed by CocoaPods) belonging to the target to the <code>Frameworks</code> folder inside the executable bundle of that target. Those frameworks were copied from the framework build products located at the same folder as the executable bundle.</p>
<pre><code class="hljs language-lua">Debug-iphonesimulator / <span class="hljs-comment">-- App.app / -- Frameworks / -- Dynamic_A.framework</span>
                       |
                       |<span class="hljs-comment">-- Dynamic_A / -- Dynamic_A.framework 👈 👈 👈</span>
</code></pre>
<p>One thing that's worth a mention is that the checksum of the framework in the framework build products (ex. <code>Debug-iphonesimulator/Dynamic_A</code>) is different from the one in the <code>Frameworks</code> folder of the app bundle. This is because CocoaPods strips some unnecessary info of frameworks while copying them to the app bundles. The stripped info includes <code>Headers</code>, <code>PrivateHeaders</code>, <code>Modules</code> folder inside the <code>.framework</code> bundle and so forth.</p>
<pre><code class="hljs language-sh"><span class="hljs-function"><span class="hljs-title">install_framework</span></span>()
{
  ...
  rsync --delete -av <span class="hljs-string">"<span class="hljs-variable">${RSYNC_PROTECT_TMP_FILES[@]}</span>"</span> --links --filter <span class="hljs-string">"- CVS/"</span> --filter <span class="hljs-string">"- .svn/"</span> --filter <span class="hljs-string">"- .git/"</span> --filter <span class="hljs-string">"- .hg/"</span> --filter <span class="hljs-string">"- Headers"</span> --filter <span class="hljs-string">"- PrivateHeaders"</span> --filter <span class="hljs-string">"- Modules"</span> <span class="hljs-string">"<span class="hljs-variable">${source}</span>"</span> <span class="hljs-string">"<span class="hljs-variable">${destination}</span>"</span>
  ...
}
</code></pre>
<h3 id="32-resourcesresource-bundles-of-static-frameworks">3.2. Resources/Resource Bundles of Static Frameworks</h3>
<p>For pods integrated as static frameworks, their resources and resource bundles will be copied to the executable bundles. And the duplication of these contents takes place the same way dynamics frameworks getting duplicated.</p>
<pre><code class="hljs language-lua">App.app / <span class="hljs-comment">-- App (*)</span>
         |
         |<span class="hljs-comment">-- StaticFW_C.bundle</span>
         |
         |<span class="hljs-comment">-- ResourcesOfStaticFW_D / -- an_image.png</span>
</code></pre>
<h2 id="4-reducing-artifacts-based-on-duplications">4. Reducing Artifacts Based on Duplications</h2>
<p>Based on the observations above, we can thin the artifacts by storing the bundle contents in a storage. Each bundle is unique by its checksum.</p>
<p><img src="/images/ci/artifacts_opt_storage.png" alt=""></p>
<p>To keep track of the original location of a bundle inside the storage, we need a mapping that maps the original location of a bundle to its place in the storage. This way, after thinning the artifacts, we can easily recover the artifacts to its original state. The contents integrity remains unchanged.</p>
<pre><code class="hljs language-lua">storage/ <span class="hljs-comment">-- &#x3C;hash_a>-A.framework</span>
        |<span class="hljs-comment">-- &#x3C;hash_b>-B.framework</span>
        |<span class="hljs-comment">-- &#x3C;hash_c>-C.framework</span>
        |<span class="hljs-comment">-- ...</span>
        |<span class="hljs-comment">-- storage.json # Contains the mapping</span>
</code></pre>
<p>With this storage mapping solution, there are 2 additional steps running on CI:</p>
<p><img src="/images/ci/artifacts_opt_build_test.png" alt=""></p>
<ul>
<li>In the <code>build</code> job, run the <strong><code>optimize</code></strong> step at the end of the job.
<ul>
<li>In this step, we look for contents under the the build products folder that match certain patterns such as <code>**/*.framework</code>, <code>**/*.bundle</code> and <code>**/*.dylib</code>. For each item, we move them to the storage (if not exist there), then add it to the mapping.</li>
</ul>
</li>
<li>In the <code>test</code> job, run the <strong><code>recover</code></strong> step at the beginning of the job.
<ul>
<li>In this step, we copy the item in the storage to its original places based on the mapping.</li>
</ul>
</li>
</ul>
<h2 id="5-discussion">5. Discussion</h2>
<p>First, the technique to remove duplications like above is nothing new, and is not specific to iOS development. It can be applied to any project. It's just that the problem becomes more noticeable with iOS projects because of the way Xcode structures the build products.</p>
<p>Second, The number of dynamic frameworks in the project plays an important role in the performance of this solution. Normally, inside a <code>.framework</code> bundle, for ex. <code>A.framework</code>, the framework binary <code>A.framework/A</code> takes up most of the space. However, in case of static frameworks, their binaries are merged into the executable binaries during the linking step (done by the <a href="https://linux.die.net/man/1/ld"><code>ld</code></a> linker). That means, we cannot reduce much for static frameworks except their resources and resource bundles.</p>
<ul>
<li>About a year ago, we maintained a duo linking strategies. If build for testing, we turn all frameworks into dynamic frameworks (except some cases). Otherwise, make them static. Making the majority of them dynamic is to resolve a code coverage issue which has been now resolved. Back then, this technique was proven to be powerful. The <em>optimize</em> step reduced the artifacts from arounnd 3GB to only 800MB. However, after shifting from dynamic frameworks to static frameworks, we have seen a decrease in the thinning performance. At the moment, the artifacts are thinned from 3.6GB to 2.1GB after the <em>optimize</em> step.</li>
</ul>
<p>In some cases, we observe 2 frameworks with the same name but having different checksums in the storage.</p>
<pre><code class="hljs language-css">storage / -- &#x3C;hash_1>-<span class="hljs-selector-tag">A</span><span class="hljs-selector-class">.framework</span>
         |-- &#x3C;hash_2>-<span class="hljs-selector-tag">A</span><span class="hljs-selector-class">.framework</span>
</code></pre>
<p>This sometimes happens when you declare pods with different forms in different targets. For example, one target is using a pod with subspec <code>A/Child_1</code> and another target is using <code>A/Child_2</code>. Another scenario this might happen is when you use a different dependencies manager that strips framework bundles differently. In such cases, you can run another round of optimization on the storage.</p>
<p>Last, does this work if we don't use CocoaPods as the dependencies manager in the project? Actually, what other dependencies managers do is similar in essence. This tip is based on the duplications (in executable bundles structure) when we have more targets in the project.
