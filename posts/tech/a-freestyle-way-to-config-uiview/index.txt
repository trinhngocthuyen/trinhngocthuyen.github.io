13:T1c12,<p>This article is to introduce a freestyle way to set up views in iOS (by code).</p>
<p>Normally, a view is set up like this</p>
<pre><code class="hljs language-swift">label <span class="hljs-operator">=</span> <span class="hljs-type">UILabel</span>()
label.text <span class="hljs-operator">=</span> <span class="hljs-string">"Hello, It's raining outside. Enjoy?"</span>
label.textAlignment <span class="hljs-operator">=</span> .<span class="hljs-type">Center</span>
label.font <span class="hljs-operator">=</span> .headerFont()
label.textColor <span class="hljs-operator">=</span> .headerColor()
view.addSubview(label)
</code></pre>
<p>Configuring a view sometimes includes a sequence of steps. We can rougly call it "chaining actions". "Chaining actions", in the general perception, is the execution of multiple actions on the same actor.</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">actor</span>.doSomething()
     .doNextStep()
     .doFinalStep()	
</code></pre>
<p>Based on that premise, we could think of this implementation</p>
<pre><code class="hljs language-swift">label <span class="hljs-operator">=</span> <span class="hljs-type">UILabel</span>()
	.config {
		<span class="hljs-variable">$0</span>.text <span class="hljs-operator">=</span> <span class="hljs-string">"Hello, It's raining outside. Enjoy?"</span>
		<span class="hljs-variable">$0</span>.textAlignment <span class="hljs-operator">=</span> .<span class="hljs-type">Center</span>
		<span class="hljs-variable">$0</span>.font <span class="hljs-operator">=</span> .headerFont()
		<span class="hljs-variable">$0</span>.textColor <span class="hljs-operator">=</span> .headerColor()
	}
	.addTo(view)
</code></pre>
<p>And</p>
<pre><code class="hljs language-swift">view.addSubview(label, withConfig: {
		<span class="hljs-variable">$0</span>.text <span class="hljs-operator">=</span> <span class="hljs-string">"Hello, It's raining outside. Enjoy?"</span>
		<span class="hljs-variable">$0</span>.textAlignment <span class="hljs-operator">=</span> .<span class="hljs-type">Center</span>
		<span class="hljs-variable">$0</span>.font <span class="hljs-operator">=</span> .headerFont()
		<span class="hljs-variable">$0</span>.textColor <span class="hljs-operator">=</span> .headerColor()
	})
	.addSubview(button, withConfig: {
		<span class="hljs-variable">$0</span>.setTitle(<span class="hljs-string">"Enjoy"</span>, forState: .<span class="hljs-type">Normal</span>)
		<span class="hljs-variable">$0</span>.addTarget(<span class="hljs-keyword">self</span>, action: <span class="hljs-keyword">#selector</span>(tapYes), forControlEvents: .<span class="hljs-type">TouchUpInside</span>)
	})
</code></pre>
<p>In fact, this style consumes more lines of code than the conventional approach. Yet, it reflects the idea more clearly. Those inside a closure <code>config</code> implicitly mean they are in the same context, making the code more structured and readable.</p>
<p>Now, let's codeeee</p>
<p>Basically, we would make some extensions over <code>UIView</code></p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">extension</span> <span class="hljs-title class_">UIView</span> {
	<span class="hljs-keyword">func</span> <span class="hljs-title function_">config</span>(<span class="hljs-meta">@noescape</span> <span class="hljs-params">closure</span>: <span class="hljs-type">UIView</span> -> <span class="hljs-type">Void</span>) -> <span class="hljs-type">UIView</span> {
        closure(<span class="hljs-keyword">self</span>)
        <span class="hljs-keyword">return</span> <span class="hljs-keyword">self</span>
    }

    <span class="hljs-keyword">func</span> <span class="hljs-title function_">addTo</span>(<span class="hljs-params">parent</span>: <span class="hljs-type">UIView</span>) -> <span class="hljs-keyword">Self</span> {
        parent.addSubview(<span class="hljs-keyword">self</span>.nt_view)
        <span class="hljs-keyword">return</span> <span class="hljs-keyword">self</span>
    }
}
</code></pre>
<p>Sadly, the closure is not generic enough to config a <code>UILabel</code> since it takes a <code>UIView</code> as the parameter
<em>(⟶ You cannot declare <code>$0.text = "..."</code> because <code>$0</code> is a <code>UIView</code>. Do not think of casting it to <code>UILabel</code> :D)</em></p>
<p>You may think of the keyword <code>Self</code> to make the compiler infer the input's type. Yes, you're on the right way.
However, it's a classic problem that <code>Self</code> is only valid for <code>protocol extensions</code>
<em>(Remember: implementing extensions with so-called generic things is kind of limited)</em></p>
<p>To overcome this issue, we create a protocol <code>NTViewType</code> which requires a properties <code>nt_view: UIView</code>. The idea is quite simple. We <strong>perform extensions on this protocol</strong> and the <strong>detail implementation operates on <code>nt_view</code></strong>. And the value <code>nt_view</code> of <code>UIView</code> is <strong>just itself</strong>.</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">protocol</span> <span class="hljs-title class_">NTViewType</span> {

<span class="hljs-keyword">extension</span> <span class="hljs-title class_">NTViewType</span> {
	<span class="hljs-keyword">func</span> <span class="hljs-title function_">config</span>(<span class="hljs-meta">@noescape</span> <span class="hljs-params">closure</span>: <span class="hljs-keyword">Self</span> -> <span class="hljs-type">Void</span>) -> <span class="hljs-keyword">Self</span> {
		closure(<span class="hljs-keyword">self</span>)
