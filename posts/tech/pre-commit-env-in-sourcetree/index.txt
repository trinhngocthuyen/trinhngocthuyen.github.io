13:T1724,<p>In the <a href="/posts/tech/pre-commit">previous post</a>, I mentioned <a href="https://pre-commit.com">pre-commit</a> as a powerful tool to lint and format in a project.</p>
<p>It had worked seamlessly for me until I committed code using SourceTree. Just to clarify, I predominantly use git on terminal. I only use a GUI app such as SourceTree to view the diff, or to stage selective chunks in a file (which is a bit difficult to achieve when using terminal). Therefore, the issue went unnoticed during my usual workflow.</p>
<h3 id="sourcetree-and-ruby-environment">SourceTree and Ruby environment</h3>
<p>The problem happened when pre-commit attempted to install Rubocop for Ruby linting, as highlighted in the error log from SourceTree.</p>
<p><img src="/images/misc/pre_commit_sourcetree_error.jpg" alt=""></p>
<p><span class="figcaption">Figure 1. Error log encountered when committing code in SourceTree.</span></p>
<p>The log revealed that it was using the System Ruby (version 2.6). This is unexpected as I manage Ruby versions with <a href="https://github.com/rbenv/rbenv">rbenv</a>. Running <code>which ruby</code> on terminal showed a different path: <code>/Users/thuyen/.rbenv/shims/ruby</code>.</p>
<h3 id="investigating-the-path">Investigating the PATH</h3>
<p>My very first doubt was that the PATH variable might not be correctly configured. To verify, I modified the pre-commit hook script (located at <code>.git/hooks/pre-commit</code>) to print the PATH value and the resolved paths of <code>ruby</code> and <code>python</code> for debugging purposes.</p>
<pre><code class="hljs language-bash"><span class="hljs-meta">#!/usr/bin/env bash</span>
<span class="hljs-comment"># File generated by pre-commit: https://pre-commit.com</span>
<span class="hljs-comment"># ID: 138fd403232d2ddd5efb44317e38bf03</span>

<span class="hljs-built_in">echo</span> <span class="hljs-variable">$PATH</span>                 <span class="hljs-comment"># &#x3C;--- INSERTED</span>
<span class="hljs-built_in">which</span> ruby &#x26;&#x26; <span class="hljs-built_in">which</span> python <span class="hljs-comment"># &#x3C;--- INSERTED</span>

...
</code></pre>
<p><img src="/images/misc/pre_commit_sourcetree_print_path.jpg" alt=""></p>
<p><span class="figcaption">Figure 2. Echoing the PATH value along with the resolved paths of <code>ruby</code> and <code>python</code>.</span></p>
<p>Meanwhile, the PATH we see when running on terminal (ie. creating a shell session) is as follows:</p>
<pre><code class="hljs language-bash">/opt/homebrew/opt/python@3.10/libexec/bin:/opt/bin:/Users/thuyen/.rbenv/shims:/opt/homebrew/bin:/opt/homebrew/sbin:/usr/local/bin:/System/Cryptexes/App/usr/bin:/usr/bin:/bin:/usr/sbin:/sbin:/Library/Apple/usr/bin:/var/run/com.apple.security.cryptexd/codex.system/bootstrap/usr/local/bin:/var/run/com.apple.security.cryptexd/codex.system/bootstrap/usr/bin:/var/run/com.apple.security.cryptexd/codex.system/bootstrap/usr/appleinternal/bin:/Users/thuyen/.rugby/clt
</code></pre>
<p>Comparing the two PATH values, we noticed that the order of <code>/Users/thuyen/.rbenv/shims</code> and <code>/usr/bin</code> was reversed in SourceTree, causing <code>ruby</code> to resolve to <code>/usr/bin/ruby</code>.</p>
<h3 id="resolving-the-issue">Resolving the issue</h3>
<p>Examining the log, we found that <code>/usr/bin/</code> follows <code>/usr/local/bin</code> in the PATH. This gives us a hint to create a symlink in <code>/usr/local/bin/ruby</code> that points to the rbenv managed version.</p>
<pre><code class="hljs language-bash">$ <span class="hljs-built_in">ln</span> -s $(<span class="hljs-built_in">which</span> ruby) /usr/local/bin/ruby
</code></pre>
<p>As Ruby related flow heavily relies on <code>gem</code> and <code>bundle</code>. It’s strongly recommended to do the same for these programs.</p>
<pre><code class="hljs language-bash">$ <span class="hljs-built_in">ln</span> -s $(<span class="hljs-built_in">which</span> gem) /usr/local/bin/gem
$ <span class="hljs-built_in">ln</span> -s $(<span class="hljs-built_in">which</span> bundle) /usr/local/bin/bundle
</code></pre>
<p>Now the hook works perfectly in SourceTree.</p>
<h3 id="discussion">Discussion</h3>
<p>Similar issues may arise with other scripting languages included in macOS, such as Python and Perl. In my case, pre-commit worked perfectly fine with Python, mainly because it executes the program <code>python</code> which is not under <code>/usr/bin</code>. However, you might see unexpected behavior when using <code>python3</code> because it’s present in <code>/usr/bin</code>, pointing to the System Python. To deal with this scenario, I’d also recommend creating a symlink <code>/usr/local/bin/python3</code> to the version you’re using (managed by Homebrew, in my case).</p>
<p>One more thing I want to point out is how the PATH was handled in SourceTree. Typically, when starting a macOS app (but not from terminal), the app doesn’t inherit environment variables from a shell session. In other words, custom variables defined in <code>~/.zshrc</code> (if your default shell is zsh, for example) will not included upon app launch. The question is how come <code>/opt/homebrew/opt/python@3.10/libexec/bin</code> appears in the PATH. SourceTree seems to modify the PATH environment variable. My hypothesis is that SourceTree initializes a shell session just for the sake of reading &#x26; updating PATH. You can test hypothesis by adding the following to <code>~/.zshrc</code>  and restarting SourceTree.</p>
<pre><code class="hljs language-bash"><span class="hljs-built_in">echo</span> $(<span class="hljs-built_in">date</span> +%s) load <span class="hljs-variable">$0</span> >> ~/Downloads/trace
</code></pre>
<p>The below entry in the trace should confirm that <code>~/.zshrc</code> was sourced during app launch.</p>
