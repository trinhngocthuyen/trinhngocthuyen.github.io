13:T4537,<p>First of all, this is not "how an iOS/MacOS app is built". An app consists of a bunch of source code files, structured in modules/frameworks, each of which could be purely in swift/objective-c, or mixed and match. Besides, linking those modules is also another aspect. <a href="https://stackoverflow.com/questions/15198725/build-or-compile">The two terms <em>compiling</em> and <em>building</em> should not be confused!</a></p>
<p>This post is about how the compiler translates a single Swift file into lower-level code. In other words, we are interested in what happens when we run this command:</p>
<pre><code class="hljs language-css">xcrun swiftc <span class="hljs-selector-tag">main</span><span class="hljs-selector-class">.swift</span>
</code></pre>
<h3 id="compilation-pipeline">Compilation pipeline</h3>
<p><img src="/images/ios/swift_build_pipeline.png" alt="">
<span class="figcaption">Source: <a href="http://llvm.org/devmtg/2015-10/slides/GroffLattner-SILHighLevelIR.pdf">http://llvm.org/devmtg/2015-10/slides/GroffLattner-SILHighLevelIR.pdf</a></span></p>
<p>[1] <strong>Parse</strong>. First, the compiler parses the source code and build the <a href="https://en.wikipedia.org/wiki/Abstract_syntax_tree">Abstract syntax tree (AST)</a>. We could see the AST by the option <code>-dump-ast</code>:</p>
<pre><code class="hljs language-css">xcrun swiftc -dump-ast <span class="hljs-selector-tag">main</span><span class="hljs-selector-class">.swift</span>
</code></pre>
<p>Semantic analysis could be performed when the AST is constructed.</p>
<p>[2] <strong>SILGen</strong>. Generate the Swift intermediate language. To get the SIL after this phase:</p>
<pre><code class="hljs language-css">xcrun swiftc -emit-silgen <span class="hljs-selector-tag">main</span><span class="hljs-selector-class">.swift</span>
</code></pre>
<p>[3] <strong>SIL Optimizations</strong>. Perform some performance optimizations on the SIL generated.</p>
<pre><code class="hljs language-css">xcrun swiftc -emit-sil <span class="hljs-selector-tag">main</span><span class="hljs-selector-class">.swift</span>
</code></pre>
<p>[4] <strong>IR</strong>. Generate the LLVM Intermediate representation. You can examine the IR by:</p>
<pre><code class="hljs language-css">xcrun swiftc -emit-ir <span class="hljs-selector-tag">main</span><span class="hljs-selector-class">.swift</span>
</code></pre>
<p>[5] <strong>Code Generation</strong>. LLVM generates the assembly code and finally produces lower-level code (.o, executable...). To view the assembly:</p>
<pre><code class="hljs language-css">xcrun swiftc -S <span class="hljs-selector-tag">main</span><span class="hljs-selector-class">.swift</span>
</code></pre>
<h3 id="swift-intermediate-language-sil">Swift Intermediate Language (SIL)</h3>
<h4 id="name-mangling">Name mangling</h4>
<p>My first look at the SIL was like "omg. what the heck is <code>_T04main6AnimalC9makeSoundyyF</code>?". But it's not as scary as you thought.</p>
<pre><code class="hljs language-perl">// Animal.makeSound()
sil hidden <span class="hljs-variable">@_T04main6AnimalC9makeSoundyyF</span> : <span class="hljs-variable">$@</span>convention(<span class="hljs-function"><span class="hljs-keyword">method</span>) (@<span class="hljs-title">guaranteed</span> <span class="hljs-title">Animal</span>) -> () </span>{
// <span class="hljs-variable">%0</span>                                             // user: <span class="hljs-variable">%1</span>
bb0(<span class="hljs-variable">%0</span> : <span class="hljs-variable">$Animal</span>):
  debug_value <span class="hljs-variable">%0</span> : <span class="hljs-variable">$Animal</span>, let, name <span class="hljs-string">"self"</span>, argno <span class="hljs-number">1</span> // id: <span class="hljs-variable">%1</span>
  <span class="hljs-variable">%2</span> = tuple ()                                   // user: <span class="hljs-variable">%3</span>
  <span class="hljs-keyword">return</span> <span class="hljs-variable">%2</span> : <span class="hljs-variable">$(</span>)                                 // id: <span class="hljs-variable">%3</span>
} // end sil function <span class="hljs-string">'_T04main6AnimalC9makeSoundyyF'</span>
</code></pre>
<p><em><strong>Name mangling</strong></em> is used to squash additional information of an entity into a single string. The encoded name could tell us its type (class/struct/enum), module, context... For example, in <code>_T04main6PersonVACycfC</code>, the letter <code>V</code> following <code>Person</code> implies that <code>Person</code> is a struct. We won't dive into the detail of this technique. For more info, you could read <a href="https://github.com/apple/swift/blob/master/docs/ABI/Mangling.rst">here</a>.</p>
<h4 id="make-sil-more-readable">Make SIL more readable</h4>
<p>We could trace a mangled string back to the originally readable text using <code>swift-demangle</code></p>
<pre><code class="hljs language-arduino">xcrun swift-demangle _T04main6AnimalC9makeSoundyyF
<span class="hljs-comment">// Output: _T04main6AnimalC9makeSoundyyF ---> main.Animal.makeSound() -> ()</span>
</code></pre>
<p>In short, more friendly SIL could be obtained by:</p>
<pre><code class="hljs language-css">xcrun swiftc -emit-silgen <span class="hljs-selector-tag">main</span><span class="hljs-selector-class">.swift</span> | xcrun swift-demangle
</code></pre>

<h4 id="a-walk-through-sil-syntax">A walk through SIL syntax</h4>
<p>Consider this simple code:</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">struct</span> <span class="hljs-title class_">Person</span> { }
<span class="hljs-keyword">class</span> <span class="hljs-title class_">Animal</span> {
  <span class="hljs-keyword">func</span> <span class="hljs-title function_">makeSound</span>() { }

<span class="hljs-keyword">func</span> <span class="hljs-title function_">isEndangered</span>(<span class="hljs-params">animal</span>: <span class="hljs-type">Animal</span>) -> <span class="hljs-type">Bool</span> { <span class="hljs-keyword">return</span> <span class="hljs-literal">false</span> }

<span class="hljs-keyword">class</span> <span class="hljs-title class_">Dog</span>: <span class="hljs-title class_ inherited__">Animal</span> {

}
</code></pre>
<p>Let's look at the SIL and demystify some basic syntax. I strongly recommend this <a href="https://github.com/apple/swift/blob/master/docs/SIL.rst">official documentation</a> for full details.</p>
<pre><code class="hljs language-perl">// Animal.makeSound()
sil hidden <span class="hljs-variable">@main</span>.Animal.makeSound() -> () : <span class="hljs-variable">$@</span>convention(<span class="hljs-function"><span class="hljs-keyword">method</span>) (@<span class="hljs-title">guaranteed</span> <span class="hljs-title">Animal</span>) -> () </span>{
// <span class="hljs-variable">%0</span>                                             // user: <span class="hljs-variable">%1</span>
bb0(<span class="hljs-variable">%0</span> : <span class="hljs-variable">$Animal</span>):
  debug_value <span class="hljs-variable">%0</span> : <span class="hljs-variable">$Animal</span>, let, name <span class="hljs-string">"self"</span>, argno <span class="hljs-number">1</span> // id: <span class="hljs-variable">%1</span>

</code></pre>
<ul>
<li>A function starts with keyword <code>sil</code>.</li>
<li>The keyword <code>hidden</code> corresponds to <code>internal</code> in Swift code, indicating that this function is only visible to objects in the same Swift module.</li>
<li><code>@main.Animal.makeSound() -> ()</code> is the demangled text of <code>_T04main6AnimalC9makeSoundyyF</code>, representing the function name.</li>
<li><code>$@convention(method)</code> means: a call to this function requires a context. For example, in <code>self.makeSound()</code>, <code>self</code> is the context of the function call.</li>
<li><code>$@convention(thin)</code> says: this is a free function. No context is needed to make an invocation.</li>
</ul>
<pre><code class="hljs language-less"><span class="hljs-selector-tag">sil</span> <span class="hljs-selector-tag">hidden</span> @<span class="hljs-selector-tag">main</span><span class="hljs-selector-class">.Dog</span><span class="hljs-selector-class">.makeFriends</span>(<span class="hljs-attribute">animal</span>: main.Animal, <span class="hljs-attribute">person</span>: main.Person) <span class="hljs-selector-tag">-</span>> () : $@<span class="hljs-selector-tag">convention</span>(method) (<span class="hljs-variable">@owned</span> Animal, Person, <span class="hljs-variable">@guaranteed</span> Dog) <span class="hljs-selector-tag">-</span>> () {

</ul>

<li>Each class has a <a href="https://en.wikipedia.org/wiki/Virtual_method_table">virtual table</a> <code>vtable</code> in order for the compiler to lookup the correct method to execute in runtime (if it's dynamically dispatched). We will talk about method dispatch in the upcoming posts.</li>
</ul>
<h3 id="conclusion">Conclusion</h3>
<p>In this post, we took a glance into the pipeline in which Swift code is compiled. We also looked into the SIL, a high-level intermediate language Apple came up with for analysis and optimization of Swift code.</p>
<p>From the perspective of a practical developer, this topic does not help us write clean code or overwhelming design patterns. Despite that, it helps you understand (just a tiny bit) what the code is actually going on under the hood, and... thereby less scared when reading lower-level interpretations of the code. By reading these stuffs, we could figure out some specific patterns that are at the core principles of the language.</p>
