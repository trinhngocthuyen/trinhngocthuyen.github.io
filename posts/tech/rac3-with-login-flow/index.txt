13:T21fc,<p>In this post, we will have a look at an example on how to use
<a href="https://github.com/ReactiveCocoa/ReactiveCocoa"><code>ReactiveCocoa</code></a> (v3.0) to handle
a simple Login workflow.</p>
<h3 id="an-example">An example</h3>
<p>First of all, you may wonder why we should use it. Let’s have a look at the
following example.</p>
<p>Almost every app needs authentication, which is simply implemented by
login with email and password. It is not only a network task, but also a
task requiring interactions with a server. But the problem is: <em>“every
task may fail”</em>. This leads to the fact that sometimes we <em>spend more time
handling failures than successful cases</em>. These failures include
network failures and server-interaction failures.</p>
<h4 id="traditional-version">Traditional version</h4>
<pre><code class="hljs language-swift"><span class="hljs-keyword">func</span> <span class="hljs-title function_">tapLoginButton</span>() {
   <span class="hljs-operator">...</span>
   <span class="hljs-type">YourAPI</span>.login(loginParameters) { (result, error) <span class="hljs-keyword">in</span>
      <span class="hljs-keyword">if</span> error <span class="hljs-operator">!=</span> <span class="hljs-literal">nil</span> {
         <span class="hljs-keyword">switch</span> error.type {
         <span class="hljs-keyword">case</span> .<span class="hljs-type">NetworkError</span>:
            <span class="hljs-comment">// Handle network failure</span>
         <span class="hljs-keyword">case</span> .<span class="hljs-type">IncorrectEmailOrPassword</span>:
            <span class="hljs-comment">// Handle failure</span>
         <span class="hljs-keyword">case</span> .<span class="hljs-type">InvalidInformation</span>:
            <span class="hljs-comment">// Handle failure</span>
            <span class="hljs-comment">// …</span>
         } <span class="hljs-keyword">else</span> {
            <span class="hljs-comment">// Handle success</span>
         }
      }
   }
}
</code></pre>
<p>This code has a few disadvantages:</p>
<ul>
<li>The handling implementation is put inside a closure. If we have 10 tasks that
need to be executed immediately after logging in, they have to be placed in this
closure. Ugly and hard to debug, right?</li>
<li>Assume we need chaining tasks: If login task is done → execute task 1. If task 1
is done → execute task 2… In that case, each task need a closure:</li>
</ul>
<pre><code class="hljs language-swift"><span class="hljs-keyword">func</span> <span class="hljs-title function_">task1</span>(<span class="hljs-params">completion</span>: (<span class="hljs-type">SuccessType</span>) -> ())
<span class="hljs-keyword">func</span> <span class="hljs-title function_">task2</span>(<span class="hljs-params">completion</span>: (<span class="hljs-type">SuccessType</span>) -> ())
</code></pre>
<ul>
<li>Errors and successful results don’t exist simultaneously. The parameters
declaration (result, error) seems redundant.</li>
</ul>
<h4 id="how-to-refactor">How to refactor?</h4>
<p>Replace closures by <code>Signals</code> or <code>SignalProducers</code>. We will discuss the
differences between <code>Signal</code> and <code>SignalProducer</code> later.</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">class</span> <span class="hljs-title class_">API</span> {
   <span class="hljs-keyword">static</span> <span class="hljs-keyword">func</span> <span class="hljs-title function_">login</span>(<span class="hljs-params">loginParameters</span>: <span class="hljs-type">LoginParameters</span>) -> 
                         <span class="hljs-type">SignalProducer</span>&#x3C;<span class="hljs-type">SuccessType</span>, <span class="hljs-type">ErrorType</span>> {
      <span class="hljs-keyword">let</span> signalProducer <span class="hljs-operator">=</span> <span class="hljs-type">SignalProducer</span>&#x3C;<span class="hljs-type">SuccessType</span>, <span class="hljs-type">ErrorType</span>> {
         sink, disposable <span class="hljs-keyword">in</span>
         <span class="hljs-comment">// For now, dont care much about `sink` and `disposable`</span>
         <span class="hljs-comment">// Send request</span>
         <span class="hljs-comment">// Validate request</span>
         <span class="hljs-keyword">if</span> networkErrorOccured() {
            <span class="hljs-keyword">let</span> error <span class="hljs-operator">=</span> makeUpNetworkError()
            sendError(sink, error)
         } <span class="hljs-keyword">else</span> <span class="hljs-keyword">if</span> serverErrorOccured() {
            <span class="hljs-keyword">let</span> error <span class="hljs-operator">=</span> makeUpServerError()
            sendError(sink, error)
         } <span class="hljs-keyword">else</span> {
            <span class="hljs-keyword">let</span> successResult <span class="hljs-operator">=</span> parseJsonAndGetSuccessResult()
            sendNext(sink, successResult)
            sendCompleted(sink)
         }
      }
   }
}
<span class="hljs-keyword">func</span> <span class="hljs-title function_">tapLoginButton</span>() {
   <span class="hljs-keyword">let</span> loginParameters <span class="hljs-operator">=</span> <span class="hljs-type">LoginParameters</span>(username, password)
   <span class="hljs-keyword">let</span> loginSignalProducer <span class="hljs-operator">=</span> <span class="hljs-type">API</span>.login(loginParameters)
   <span class="hljs-comment">// Task 1</span>
   loginSignalProducer
   <span class="hljs-operator">|></span> start(error: { error <span class="hljs-keyword">in</span>
      handleErrorTask1()
   }, next { successfulResult <span class="hljs-keyword">in</span>
      handleSuccessTask1()
   })
   <span class="hljs-comment">// Task 2</span>
   loginSignalProducer
   <span class="hljs-operator">|></span> observe(error: { error <span class="hljs-keyword">in</span>
      handleErrorTask2()
   }, next { successfulResult <span class="hljs-keyword">in</span>
      handleSuccessTask2()
   })
   <span class="hljs-operator">…</span>
   <span class="hljs-comment">// Task 10</span>
   loginSignalProducer
   <span class="hljs-operator">|></span> observe(error: { error <span class="hljs-keyword">in</span>
      handleErrorTask10()
   }, next { successfulResult <span class="hljs-keyword">in</span>
      handleSuccessTask10()
   })
}
</code></pre>
<p>This implementation looks more elegant since:</p>
<ul>
<li>Failure and success are handled separately</li>
<li>Each observation is handled separately</li>
<li>We dont have to take much care of asynchronous tasks</li>
</ul>
<h3 id="what-makes-differences">What makes differences?</h3>
<p>I think what make sense are the abstract types:</p>
<ul>
<li><code>Result</code>: handles failures</li>
<li><code>Event</code>: handles asynchronous tasks</li>
<li><code>Signal</code>: handles observation for changes along with time</li>
</ul>
<h4 id="a-little-explanation">A little explanation</h4>
<p>If you already heard of <a href="https://www.coursera.org/course/reactive">FRP (Functional Reactive
Programming)</a>, this may help you
understand more straightforwardly:</p>
<pre><code class="hljs language-swift"><span class="hljs-operator">-</span> <span class="hljs-type">Result</span>&#x3C;<span class="hljs-type">SuccessType</span>, <span class="hljs-type">ErrorType</span>>  <span class="hljs-operator">=</span> <span class="hljs-type">Try</span>[<span class="hljs-type">SuccessType</span>, <span class="hljs-type">ErrorType</span>]
<span class="hljs-operator">-</span> <span class="hljs-type">Event</span>&#x3C;<span class="hljs-type">SomeType</span>, <span class="hljs-type">ErrorType</span>>      <span class="hljs-operator">=</span> <span class="hljs-type">Future</span>[<span class="hljs-type">SomeType</span>, <span class="hljs-type">ErrorType</span>]
<span class="hljs-operator">-</span> <span class="hljs-type">Signal</span>&#x3C;<span class="hljs-type">SomeType</span>, <span class="hljs-type">ErrorType</span>>     <span class="hljs-operator">=</span> <span class="hljs-type">Observable</span>[<span class="hljs-type">SomeType</span>, <span class="hljs-type">ErrorType</span>]
                                  <span class="hljs-operator">=</span> a series of <span class="hljs-type">Events</span>
</code></pre>
<h3 id="conclusion">Conclusion</h3>
<ul>
<li>Don’t waste your time implementing Observer pattern or manually handling
asynchronous tasks.</li>
<li>If you’re in favor of Java, a similar framework could be found as
<a href="https://github.com/ReactiveX/RxJava">RxJava</a>.</li>
</ul>
