13:T11b1,<p>Lazy evaluation is a really powerful technique which enhances app performance by avoiding unecessary computation. In Swift, apart from this advantage, it also brings convenience when it comes to coding style. For instance, I usually place the setup of a UI element in a closure instead of putting a whole bunch of code in <code>viewDidLoad()</code>. And when we need <code>self</code> for that setup, <code>lazy</code> comes to the rescue.</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">private</span> <span class="hljs-keyword">lazy</span> <span class="hljs-keyword">var</span> nameLabel: <span class="hljs-type">UILabel</span> <span class="hljs-operator">=</span> {
	<span class="hljs-keyword">let</span> label <span class="hljs-operator">=</span> <span class="hljs-type">UILabel</span>()
	label.text <span class="hljs-operator">=</span> <span class="hljs-keyword">self</span>.person.name
	<span class="hljs-comment">// ... Further setup</span>
	<span class="hljs-keyword">return</span> label
}()
</code></pre>
<p>Putting aside the overhead complexity it adds to the compilation process, there are some more cases we need to pay attention.</p>
<p>Consider this code:</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">final</span> <span class="hljs-keyword">class</span> <span class="hljs-title class_">ViewController</span> {
	<span class="hljs-keyword">private</span> <span class="hljs-keyword">lazy</span> <span class="hljs-keyword">var</span> progressView: <span class="hljs-type">ProgressView</span> <span class="hljs-operator">=</span> {
		<span class="hljs-keyword">let</span> _progressView <span class="hljs-operator">=</span> <span class="hljs-type">ProgressView</span>(delegate: <span class="hljs-keyword">self</span>)
		<span class="hljs-comment">// Further setup</span>
		<span class="hljs-keyword">return</span> _progressView
	}()
	<span class="hljs-operator">...</span>
}

<span class="hljs-keyword">extension</span> <span class="hljs-title class_">ViewController</span>: <span class="hljs-title class_ inherited__">ProgressViewDelegate</span> {
	<span class="hljs-keyword">func</span> <span class="hljs-title function_">progressViewUpdateProgress</span>() {
		<span class="hljs-keyword">let</span> value <span class="hljs-operator">=</span> progressView.value
		<span class="hljs-operator">...</span>
	}
}
</code></pre>
<p>Everything seems pretty normal. But taking a closer look, we notice that there's a chance that while <code>progressView</code> is being configured, it may be called which ends up executing the closure twice. This introduces some critical issues as the second call will overwrite the value in the first setup.</p>
<pre><code class="hljs language-swift"><span class="hljs-comment">// 1. execute closure</span>
<span class="hljs-comment">// --> create, not yet return </span>
<span class="hljs-comment">// --> still marked progressView as "not yet initialized"</span>
<span class="hljs-keyword">let</span> _progressView <span class="hljs-operator">=</span> <span class="hljs-type">ProgressView</span>(delegate: <span class="hljs-keyword">self</span>) 

<span class="hljs-comment">// 2. delegate is called</span>
progressViewUpdateProgress() 

<span class="hljs-comment">// 3. access progressView, but it is marked as "not yet initialized" since we didn't reach the return</span>
<span class="hljs-comment">// --> execute closure (again)</span>
<span class="hljs-keyword">let</span> value <span class="hljs-operator">=</span> progressView.value 
</code></pre>
<p>This twice execution can lead to a few issues as follows:</p>
<ul>
<li>The creation may take forever, then the app crashes.</li>
<li>The second <code>progressView</code> is not set up properly. For example, if we first use <code>progressView</code> in <code>view.addSubview(progressView)</code>, the first one is added to <code>view</code> but the second one does not have <code>superview</code> at all 😱. The assumption that <code>progressView</code> has <code>superview</code> is not true anymore.</li>
</ul>
<p>...</p>
<p>Apparently, this should not be a limitation of the language. Rather, we should pay careful attention to such scenarios...
However, by using <code>lazy var</code>, the problem seems to hide quite cleverly.</p>
<p>In this example, we use <code>lazy var</code> just for convenience. But as a tradeoff, the app is more vulnerable. So... Better change it to <code>let</code> to make things less unpredictable...</p>
