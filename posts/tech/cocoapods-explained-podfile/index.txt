13:T2312,<p>In iOS development, CocoaPods has become one of the must-know tools. In a team where CocoaPods is adopted, engineers run <code>pod install</code> as often as <code>git checkout</code>. They know how to properly set up a project with CocoaPods, declare dependencies in Podfile, and update dependencies if needed. However, some have little understanding of what happens under the hood. This prevents them from troubleshooting CocoaPods issues if occurred or extending its usage. The “CocoaPods Explained” series I am writing aims to fill that gap.</p>
<p>In this post, we take a look at Podfile and try to understand it a little bit more.</p>
<h3 id="declaration-in-podfile">Declaration in Podfile</h3>
<p>Podfile is where we declare dependencies of the project. This file is just a snippet of ruby code on top of provided methods (reference: <a href="https://guides.cocoapods.org/syntax/podfile.html">Podfile syntax</a>).</p>
<pre><code class="hljs language-ruby">platform <span class="hljs-symbol">:ios</span>, <span class="hljs-string">"14.0"</span>
use_frameworks!

target <span class="hljs-string">"App"</span> <span class="hljs-keyword">do</span>
  pod <span class="hljs-string">"RxSwift"</span>
  <span class="hljs-comment"># ...</span>
<span class="hljs-keyword">end</span>
</code></pre>
<p>In Podfile, we declare a dependency using the <code>pod</code> function. Such a dependency could be from CocoaPods-managed repos, a git repo, or a local directory:</p>
<pre><code class="hljs language-ruby">pod <span class="hljs-string">"RxSwift"</span>, <span class="hljs-string">"6.1.0"</span>
pod <span class="hljs-string">"Moya"</span>, <span class="hljs-symbol">:git</span> => <span class="hljs-string">"https://github.com/Moya/Moya.git"</span>, <span class="hljs-symbol">:tag</span> => <span class="hljs-string">"15.0.3"</span>
pod <span class="hljs-string">"Services"</span>, <span class="hljs-symbol">:path</span> => <span class="hljs-string">"LocalPods/Services"</span>
</code></pre>
<h3 id="when-is-podfile-loaded">When is Podfile loaded?</h3>
<p>Not all pod commands load this Podfile. We can easily verify this hypothesis by printing a log message at the beginning of Podfile (ex. prepending this line <code>puts "Load Podfile"</code>). The message appears in the logs when running <code>pod install</code> while it does not when running some other commands like <code>pod repo list</code>.</p>
<p>This behavior makes sense because Podfile is meant for dependencies declaration. CLI usages related to dependencies should load Podfile. Otherwise, loading Podfile is not necessary. Looking at the pod subcommand descriptions (ie. running <code>pod --help</code>), one may intuitively tell that <code>pod install</code>, <code>pod update</code>, and <code>pod outdated</code> do load Podfile.</p>
<p><img src="/images/ios/cli_pod_help.png" alt=""></p>
<p>Looking at the code of <a href="https://github.com/CocoaPods/CocoaPods">cocoapods</a> and <a href="https://github.com/CocoaPods/Core">cocoapods-core</a>, Podfile is loaded when we first access <code>Pod::Config.instance.podfile</code> (ie. the attribute <code>podfile</code> of the config singleton).</p>
<p><a href="https://github.com/CocoaPods/CocoaPods/blob/1.11.0/lib/cocoapods/config.rb#L204-L206">https://github.com/CocoaPods/CocoaPods/blob/1.11.0/lib/cocoapods/config.rb#L204-L206</a></p>
<pre><code class="hljs language-ruby"><span class="hljs-keyword">module</span> <span class="hljs-title class_">Pod</span>
  <span class="hljs-keyword">class</span> <span class="hljs-title class_">Config</span>
    <span class="hljs-keyword">def</span> <span class="hljs-title function_">podfile</span>
	    <span class="hljs-variable">@podfile</span> |<span class="hljs-params"></span>|= <span class="hljs-title class_">Podfile</span>.from_file(podfile_path) <span class="hljs-keyword">if</span> podfile_path
    <span class="hljs-keyword">end</span>
    ...
 <span class="hljs-keyword">end</span>
<span class="hljs-keyword">end</span>
</code></pre>
<p>We can verify this easily by executing a one-line ruby code that accesses the value. The log <code>Load Podfile</code> should appear in the console.</p>
<pre><code class="hljs language-bash">$ ruby -e <span class="hljs-string">'require "cocoapods"; Pod::Config.instance.podfile'</span>

Load Podfile
</code></pre>
<h3 id="methods-in-podfile">Methods in Podfile</h3>
<p>Podfile content is <a href="https://github.com/CocoaPods/Core/blob/1.11.0/lib/cocoapods-core/podfile.rb#L327-L338">evaluated within <code>Pod::Podfile</code> context</a>. This gives us code access to <code>Pod::Podfile</code>'s instance variables and private methods. Therefore, what we can do in Podfile is much more than just dependencies declaration.</p>
<p>Have you ever got annoyed by the Podfile checksum changing all the time? This change causes a high chance of merge conflicts in Podfile.lock, especially when there are many contributors to the project.</p>
<pre><code class="hljs language-diff"><span class="hljs-deletion">-PODFILE CHECKSUM: ad5874f24bb0dbffd1e9bb8443604e3578796c7a</span>
<span class="hljs-addition">+PODFILE CHECKSUM: ab71b2c6800abc9eba8a3500b110eccf4e33e7f4</span>
</code></pre>
<p>We can tackle this problem by setting the <a href="https://github.com/CocoaPods/Core/blob/1.11.0/lib/cocoapods-core/podfile.rb#L259-L268"><code>@checksum</code> attribute</a> right in Podfile. After running <code>pod install</code>, we should see the dummy checksum being reflected in Podfile.lock.</p>
<pre><code class="hljs language-ruby"><span class="hljs-comment"># In Podfile</span>
<span class="hljs-variable">@checksum</span> = <span class="hljs-string">"dummy-text-to-reduce-merge-conflicts"</span>
</code></pre>
<p>...</p>
<p>As mentioned, we have access to <a href="https://www.rubydoc.info/gems/cocoapods-core/Pod/Podfile"><code>Pod::Podfile</code></a>'s methods in Podfile. Those commonly used in Podfile are <code>use_frameworks!</code>, <code>platform :ios, "14.0"</code>, <code>target "..."</code>, <code>pod "..."</code>. In fact, those methods are <a href="https://github.com/CocoaPods/Core/blob/1.11.0/lib/cocoapods-core/podfile/dsl.rb#L35">defined in <code>Pod::Podfile::DSL</code></a> and <a href="https://github.com/CocoaPods/Core/blob/1.11.0/lib/cocoapods-core/podfile.rb#L16">included in <code>Pod::Podfile</code></a>.</p>
<p>We can patch those methods to add more customizations. This could be done using <code>alias_method</code> with the corresponding method in <code>Pod::Podfile</code> or <code>Pod::Podfile::DSL</code>. Using <code>alias_method</code> is similar to the <a href="/posts/tech/method-swizzling-what-why-and-how/">method swizzling technique in Swift</a>. Some CocoaPods plugins alter CocoaPods’s behaviors this way to interfere with the pod installation process.</p>
<p>The following code demonstrates how we introduce the <code>:xcode_migration</code> option to the <code>pod</code> syntax. This option helps eliminate many if/else checks in Podfile, making the usage more declarative.</p>
<pre><code class="hljs language-ruby"><span class="hljs-comment"># In Podfile</span>
require_relative <span class="hljs-string">"patch"</span> <span class="hljs-comment"># &#x3C;-- Load patch.rb</span>
...
pod <span class="hljs-string">"RxSwift"</span>, <span class="hljs-symbol">:xcode_migration</span> => {
  <span class="hljs-symbol">:default</span> => <span class="hljs-string">"6.1.0"</span>,
  <span class="hljs-symbol">:xcode14</span> => {
    <span class="hljs-symbol">:git</span> => <span class="hljs-string">"link/to/fork"</span>,
    <span class="hljs-symbol">:branch</span> => <span class="hljs-string">"xcode14"</span>
  }
}
</code></pre>
<pre><code class="hljs language-ruby"><span class="hljs-comment"># In patch.rb</span>
<span class="hljs-keyword">module</span> <span class="hljs-title class_">Pod</span>
  <span class="hljs-keyword">class</span> <span class="hljs-title class_">Podfile</span>
	  alias_method <span class="hljs-symbol">:original_pod</span>, <span class="hljs-symbol">:pod</span>
	  <span class="hljs-keyword">def</span> <span class="hljs-title function_">pod</span>(<span class="hljs-params">*args, **kwargs</span>)
      <span class="hljs-keyword">if</span> (opts = kwargs[<span class="hljs-symbol">:xcode_migration</span>])
        <span class="hljs-comment"># Process opts &#x26; update args/kwargs to use the version/branch accordingly</span>
        ...
        <span class="hljs-keyword">if</span> xcode14?
          ...
        <span class="hljs-keyword">end</span>
			<span class="hljs-keyword">end</span>
	    original_pod(*args, **kwargs) <span class="hljs-comment"># &#x3C;-- call the original method with the processed args/kwargs</span>
		<span class="hljs-keyword">end</span>
  <span class="hljs-keyword">end</span>
<span class="hljs-keyword">end</span>
</code></pre>
