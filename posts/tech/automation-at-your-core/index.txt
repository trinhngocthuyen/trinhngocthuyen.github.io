13:T2112,<p>We often hear that automating repetitive tasks improves productivity. But how do we decide when automation is truly worth the effort?</p>
<p><span class="point-badge">1</span> At OKX, while managing end-to-end testing servers, I frequently needed to SSH into servers to diagnose issues and retrieve relevant logs (to my local) for further investigations. This was tedious but manageable, thanks to my terminal's auto-completion. However, as we expanded to four servers (two in Hong Kong, two in Singapore), keeping track of server IPs, usernames and passwords for the SSH became a hassle. That's when I built a small utility to simplify the process.</p>
<p>With this tool, logging into a Hong Kong server (without specifying password) was simply as:</p>
<pre><code class="hljs language-bash">$ sshe login hk
</code></pre>
<p>And pulling a certain file from a server was just:</p>
<pre><code class="hljs language-bash">$ sshe pull hk path/to/log
</code></pre>
<p>This significantly improved efficiency and freed up my time for more important tasks.</p>
<p><span class="point-badge">2</span> Beyond corporate work, I also maintain several open-source projects on Github, including some Python packages and CocoaPods plugins. From the start, I used Github actions to deploy a new release. All I had to do is to trigger a workflow under the Github actions dashboard. This worked well when managing just one or two projects. However, as the number of repos grew, synchronizing changes across them became a maintenance burden. To reduce this overhead, I extracted the deployment workflow into a shared repo: <a href="https://github.com/trinhngocthuyen/gh-actions/blob/main/.github/workflows/rb-deploy.yml">gh-actions</a>. However, the need for maintenance didn't end there.</p>
<p><img src="/images/misc/automation_gh_repos.png" alt="">
<span class="figcaption">Figure. Many repos having the same deployment workflow.</span></p>
<p>Technically, deploying a CocoaPods plugin (which is a Ruby gem) requires 2 tokens:</p>
<ul>
<li>A <a href="http://RubyGems.org">RubyGems.org</a> token to publish the gem.</li>
<li>A Github deployment token to create a release and bump the version.</li>
</ul>
<p><img src="/images/misc/automation_gh_secrets.png" alt=""></p>
<p>Managing these tokens manually seemed fine… until the token expired. I had to regenerate a new token and update the corresponding secrets across all repos above. This manual process was tedious, requiring navigation through different Github settings (Repo → Settings → Secrets and variables → Actions → make update). Once, my token expired while I was maintaining two active repos (<a href="https://github.com/trinhngocthuyen/cocoapods-spm">cocoapods-spm</a> and <a href="https://github.com/trinhngocthuyen/e2e-mobile">e2e-mobile</a>). I updated the token only for these two, assuming that I wouldn't need it for others (because I won't have any updates for them). This was such a mistake 😅! Later, when I had to update another plugin, the deployment went broken which was frustrating and costed me preliminary investigations.</p>
<p>I then wrote a CLI tool called <code>sauron</code> (the name was from LOTR) to help with some admin tasks. Thanks to this tool, I can update tokens across multiple repos matching a pattern. This can be done at hand with <a href="https://cli.github.com/manual/">Github CLI</a> or with Github APIs.</p>
<pre><code class="hljs language-bash">$ sauron gh update-token --pattern <span class="hljs-string">'cocoapods-*'</span>
</code></pre>
<p><img src="/images/misc/automation_sauron.png" alt=""></p>
<p>Of course setting a long-lived token (ex. without expiration) could have avoided this issue. But automation offers an effortless setup for new repos.</p>
<p><span class="point-badge">3</span> Automation isn't just about scripting repetitive tasks - it's also about designing workflows and orchestrating them together. Back to my deployment story, although the process was automated, it still required a manual trigger. After fixing bugs in <a href="https://github.com/trinhngocthuyen/cocoapods-spm">cocoapods-spm</a>, for example, I had to manually run the <a href="https://github.com/trinhngocthuyen/cocoapods-spm/actions/workflows/deploy.yml">deploy workflow</a> to release a new version. This is where human mistakes jump in… coz I sometimes forgot 😅.</p>
<p>To prevent it, I introduced a weekly auto-deployment workflow:</p>
<ul>
<li><a href="https://github.com/trinhngocthuyen/cocoapods-spm/actions/workflows/milestone-assignment.yml">Milestone assignment</a>: For each project, we have an active milestone (ex. 1.0.2) that indicates the yet-to-be-released version. When a PR gets merged, it is assigned to this milestone.</li>
<li><a href="https://github.com/trinhngocthuyen/cocoapods-spm/actions/workflows/weekly-auto-deploy.yml">Scheduled deployment</a>: A cron job runs every Sunday, checking if the active milestone has any associated PRs. If so, it triggers a deployment.</li>
<li><a href="https://github.com/trinhngocthuyen/cocoapods-spm/actions/workflows/milestone-creation.yml">Milestone management</a>: After a release, the workflow closes the current milestone and creates a new milestone for the next version.</li>
</ul>
<p><span class="point-badge">4</span> Another automation win came from solving LeetCode problems. I prefer coding locally for better debugging experiences. Initially, each solution file was isolated. I needed to write a bunch of boilerplate code to test against given test cases. The unrelated code was even more for graph/tree problems.</p>

<p><span class="point-badge">5</span> Returning to our original question: <em>When is the right time to automate?</em> The answer varies based on effort versus benefit. At the end of the day, it's the ROI (return on investment) that drives the actions.</p>
<p>For me, since I already work closely with automated processes, the time and effort spent setting up a certain automation task might be relatively low compared to an iOS engineer who doesn't regularly deal with it.</p>
<p>A good rule of thumb is to consider repetition:</p>
<ul>
<li>If a manual task happens several times a week, automation is likely worth it.</li>
<li>If a task occurs rarely for one, say once every 2 months, but is distributed among a team (ex. each takes turn to do it weekly), automating it might still provide significant savings as a whole.</li>
</ul>
<p>Of course, the choice also depends on other factors such as capacity and priority.</p>
