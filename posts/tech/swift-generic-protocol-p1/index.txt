<h3 id="solution">Solution</h3>
<p>Here comes the trick to overcome this problem. The idea is simple, we use a generic class to wrap the protocol.</p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">class</span> <span class="hljs-title class_">GenericDataFetcher</span>&#x3C;<span class="hljs-type">DT</span>>: <span class="hljs-title class_ inherited__">DataFetchingType</span> {
    <span class="hljs-keyword">typealias</span> <span class="hljs-type">DataType</span> <span class="hljs-operator">=</span> <span class="hljs-type">DT</span>
    <span class="hljs-operator">............</span>
}

<span class="hljs-keyword">class</span> <span class="hljs-title class_">ViewModel</span>&#x3C;<span class="hljs-type">DataType</span>> {
    <span class="hljs-keyword">var</span> fetcher: <span class="hljs-type">GenericDataFetcher</span>&#x3C;<span class="hljs-type">DataType</span>>
}

</code></pre>
<p>It's still generic and the compiler would not complain. And the hard part remaining is to make <code>GenericDataFetcher</code> flexible that we WOULD NOT write <code>EventFetcher</code> as a subclass of <code>GenericDataFetcher</code> (subclassing is not really a good idea here).</p>
<p>A solution: <em><strong>init GenericDataFetcher from a protocol, store the required functions, properties of that protocol.</strong></em></p>
<pre><code class="hljs language-swift"><span class="hljs-keyword">class</span> <span class="hljs-title class_">GenericDataFetchingType</span>&#x3C;<span class="hljs-type">DT</span>>: <span class="hljs-title class_ inherited__">DataFetchingType</span> {

}
